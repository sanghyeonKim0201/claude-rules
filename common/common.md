# 공통 규칙

**모든 답변은 한국어로 작성하세요.**

## Git 컨벤션

**브랜치 전략:**

- `live` 브랜치(기본 브랜치)에는 **직접 push 금지**. 반드시 `dev` 브랜치에 push 후 GitHub에서 PR merge를 통해 반영한다.
- `dev` 브랜치에서 개발·테스트 후, `dev → live` PR을 생성하여 merge한다.

커밋 메시지 prefix:

- `feat` — 새로운 기능 추가
- `fix` — 버그 수정
- `docs` — 문서의 수정
- `style` — (코드의 수정 없이) 스타일(style)만 변경 (들여쓰기 같은 포맷이나 세미콜론을 빼먹은 경우)
- `refactor` — 코드를 리팩토링
- `test` — Test 관련한 코드의 추가, 수정
- `chore` — (코드의 수정 없이) 설정을 변경
- `wip` — 작업 중 (Work In Progress)

**버전 관리 (태그 + 릴리즈):**

- 커밋 후 사용자가 버전 관리를 요청하면 **반드시** git 태그를 생성하고 GitHub Release를 추가한다.
- 태그는 [Semantic Versioning](https://semver.org/lang/ko/) 규칙을 따른다: `vMAJOR.MINOR.PATCH`
  - `MAJOR` — 호환되지 않는 API 변경 (기존 prop 제거, 토큰 이름 변경 등)
  - `MINOR` — 하위 호환 기능 추가 (새 컴포넌트, 새 prop 등)
  - `PATCH` — 하위 호환 버그 수정
- 태그 생성: `git tag -a vX.Y.Z -m "메시지"` → `git push origin vX.Y.Z`
- GitHub Release: `gh release create vX.Y.Z --title "제목" --notes "변경 내역"` 로 생성한다.
- Release notes에는 **신규 기능 / 기존 변경 / 버그 수정 / 주의사항**을 구분해 작성한다.

## 외부 도구 사용 전략

MCP는 AI 대화 중 자동 연동이 필요한 것만 연결하고, 나머지는 CLI로 직접 호출한다.

- **MCP 연결:** Linear, Context7
- **CLI 사용:** `gh` (GitHub), `sentry` (Sentry), `ssh` (Jenkins CLI), `doppler` (Doppler), `obsidian` (Obsidian)

## 설계 원칙

모든 코드를 설계할 때 다음 원칙을 고려한다:

- **확장성**: 새로운 요구사항이 추가될 때 기존 코드를 최소한으로 수정하도록 설계한다.
- **재사용성**: 범용적인 로직은 공유 가능한 형태로 분리한다.
- **낮은 결합도**: 모듈 간 직접적인 의존을 줄이고, 인터페이스(public API)를 통해 소통한다.
- **높은 응집도**: 관련된 코드는 한 곳에 모으고, 하나의 모듈은 하나의 명확한 책임만 갖는다.
- **관심사 분리**: UI, 비즈니스 로직, 데이터 접근을 명확히 나누어 각 레이어가 독립적으로 변경 가능하도록 한다.
- **Early Return**: 중첩 `if-else` 대신 조건 불충족 시 즉시 반환(guard clause)하여 들여쓰기와 인지 복잡도를 줄인다.
- **매직 넘버·문자열 금지**: 의미 없는 리터럴(`10`, `'ACTIVE'` 등)은 이름 있는 상수로 추출한다.
- **불변성 우선**: 배열·객체를 직접 변경하지 않고 새 참조를 생성한다.
- **조기 추상화 금지 (AHA)**: 2~3회 반복되기 전까지는 추상화하지 않는다. 비슷해 보이는 코드도 실제 변경 방향이 다를 수 있다.
- **일관성(Consistency)**: 같은 문제는 같은 방식으로 해결한다. 네이밍 패턴, 상태 접근 방식 등 프로젝트 전반에 걸쳐 동일한 컨벤션을 적용한다. 코드를 작성할 때 기존 코드의 패턴을 먼저 확인하고 그에 맞춘다.

## 네이밍 원칙

- **변수명**: **명사**로 표기한다. (예: `count`, `patientName`)
- **함수명**: **동사** 또는 **동사구**로 표기한다. (예: `findCamera()`, `getPatientData()`)
- **상수명**: **SCREAMING_SNAKE_CASE**(대문자 언더스코어)를 사용한다. (예: `MAX_RETRY_COUNT`, `API_BASE_URL`)
- **의미 중심 명명**: 축약어 사용을 지양하고, 의미를 알기 쉽게 풀어써서 명명한다.

## 순수 함수 분리 원칙

- **검증(validation)은 순수 함수로 분리**한다. 비즈니스 로직 안에 검증을 섞지 않고, 별도 함수로 추출한다.
- **데이터 변환(transformation)은 순수 함수로 분리**한다. 입력 데이터 → API 페이로드 변환 등은 순수 함수로 정의한다.
- 비즈니스 로직과 무관한 **순수 함수**(데이터 변환, 필터링, 포맷팅 등)는 호출부 외부에 별도로 정의한다.
