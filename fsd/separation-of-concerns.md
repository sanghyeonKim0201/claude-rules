# FSD 관심사 분리 원칙

FSD 아키텍처에서의 레이어·세그먼트 수준 관심사 분리 규칙이다.

이 문서는 특정 언어와 프레임워크의 문법이 아니라, **레이어 책임과 조합 방향**을 정의한다.

## 아키텍처·폴더 수준

- **FSD 레이어 간 책임**: `shared`(범용) → `entities`(도메인 모델) → `features`(사용자 액션) → `widgets`(복합 조합) → `views/pages`(화면 진입점) 순으로 의존한다.
- **같은 레이어의 slice끼리는 직접 결합하지 않는다.** 같은 레이어 간 조합이 필요하면 우선 상위 레이어로 올린다.
- **같은 레이어 간 조합이 필요한 경우**: 우선 **상위 레이어(widgets 또는 pages/views)** 에서 조합한다.
- **slot 또는 확장 포인트가 필요한 경우**: 상위 레이어의 요소를 하위 레이어 UI 내부 특정 위치에 삽입해야 할 때만 연다. 외부에서 나란히 조합 가능한 경우에는 slot보다 상위 레이어 조합이 우선이다.
- **widgets 레이어 사용 기준**: 여러 features·entities가 조합되는 복합 UI 블록이면서 **여러 화면에서 재사용**되는 경우 widgets로 분리한다. 한 화면에서만 쓰는 조합은 views/pages에서 직접 구현한다.

## 세그먼트별 역할

하나의 세그먼트가 다른 세그먼트의 책임을 침범하지 않는다.

- `ui/` — UI 컴포넌트, 프레젠테이션 로직, 스타일
- `api/` — 외부 시스템 통신, request/response mapping, data source adapter
- `model/` — 도메인 모델, 상태, 비즈니스 규칙, use case
- `lib/` — 해당 slice 내부에서 공통으로 쓰이는 유틸리티
- `config/` — 기능 설정, 환경별 설정, 상수 구성

세그먼트 이름은 프로젝트 특성에 따라 달라질 수 있지만, 책임 분리는 동일해야 한다.
