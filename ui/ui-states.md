# UI 상태 설계

**모든 UI는 다음 상태를 명시적으로 설계한다.** 정상 상태만 그리고 끝내지 않는다. 상태 누락은 UX 실패의 가장 흔한 원인.

## 필수 상태 체크리스트

데이터·인터랙션을 다루는 모든 컴포넌트는 아래 상태를 **모두 고려**한다.

| 상태 | 언제 | 필수 처리 |
|---|---|---|
| **Idle / Default** | 초기·정상 | 기본 레이아웃 |
| **Loading** | 데이터 요청 중 | 스켈레톤 / 스피너 / progressive |
| **Empty** | 데이터 없음 | 안내 문구 + 다음 액션 유도 |
| **Error** | 요청·처리 실패 | 에러 설명 + 복구 수단 (재시도 등) |
| **Disabled** | 사용 불가 | 이유 명시 (왜 비활성인가) |
| **Success** | 작업 완료 | 피드백 (토스트·인라인) |

> **설계 시 각 상태의 목업을 전부 그려보는 것**이 원칙. 개발 중 "아 이 상태 안 그렸네"를 방지.

## Loading

- **즉시 피드백**: 200ms 이상 걸리는 작업은 반드시 로딩 표현.
- **스켈레톤 우선**: 최종 레이아웃과 유사한 구조. 스피너는 영역이 작거나 구조 예측 불가할 때만.
- **레이아웃 시프트 방지**: 로딩 → 로드 완료 시 크기/위치가 크게 변하지 않도록.
- **점진적 표시 (progressive)**: 가능한 부분부터 먼저 렌더. 전체 블록킹 금지.

## Empty

- **단순 "데이터 없음"은 실패**. 사용자가 다음에 뭘 해야 할지 알려준다.
- 구성: **아이콘/일러스트 + 설명 + 액션 버튼** (선택)
- 필터링 결과가 empty면 **필터 초기화** 액션 제공.
- 최초 사용 시 empty는 **온보딩**으로 활용.

```
// ✅ 예시
"아직 등록된 프로젝트가 없어요"
→ [새 프로젝트 만들기] 버튼
```

## Error

- 사용자가 **무엇이 왜 실패했고, 어떻게 복구**할 수 있는지 명시.
- 기술적 에러 메시지 노출 금지. (`500 Internal Server Error` 등)
- **복구 수단 필수**: 재시도, 돌아가기, 고객 지원 연결 중 최소 하나.
- 전체 페이지 에러 vs 부분 에러를 구분. 부분 에러로 처리 가능하면 전체 페이지를 깨지 않는다.

```
// ❌
"Error: Request failed with status code 500"

// ✅
"주문 정보를 불러오지 못했어요. 잠시 후 다시 시도해 주세요."
[다시 시도]
```

## Disabled

- 비활성 상태는 **왜 비활성인지** 툴팁·인라인 안내로 이유를 제공한다.
- 비활성 원인이 해결 가능하면 **해결 방법**을 함께 안내.
- 시각적으로 명확히 구분 (대비 낮추되 WCAG 준수).
- `aria-disabled` 또는 `disabled` 속성으로 보조 기술에도 전달.

```
// ❌ 이유 없이 회색 처리
[ 저장 ]  (disabled)

// ✅
[ 저장 ]  (disabled)
└ "필수 항목을 모두 입력해주세요"
```

## Success

- 조용히 성공한 작업은 **명시적 피드백**이 필요. 사용자는 성공 여부를 확신하고 싶다.
- 방식:
  - 짧은 작업: 토스트·인라인 체크
  - 긴 작업: 결과 화면 전환 + 요약
- 토스트는 3~5초 후 자동 닫힘 + 접근성 라이브 영역 적용.

## 상태 간 전환

- 상태 전환은 **부드럽게**. 급격히 레이아웃이 바뀌면 혼란.
- Loading → Error 전환 시 사용자가 무엇이 바뀌었는지 인지할 수 있어야 함.
- 낙관적 업데이트(optimistic update) 사용 시 실패 복구 시나리오 반드시 설계.

## 부분 vs 전체 상태

- 가능한 한 **부분 상태**로 처리. 페이지 하나에 에러 하나라고 전체를 날리지 않는다.
- 데이터 소스가 여러 개면 **각각 독립적으로** 상태 관리.

## 금지

- **로딩 상태 없이** 비동기 작업 수행 (버튼 눌렀는데 반응 없는 것처럼 보임).
- **Empty 상태를 "No data"로만** 처리 (다음 액션 제안 없음).
- **에러를 콘솔에만** 남기고 UI에 안 알림.
- **비활성화 이유 불명**.
- **성공 피드백 생략** (특히 폼 제출, 설정 변경).
